# API Usage Map

> Generated by tracing each screen through its full chain: **screen → cubit → use case → repository → API method → endpoint**.
> Source of truth for endpoints: [`lib/core/network/api_url.dart`](../lib/core/network/api_url.dart) (`ApiUrl`).
> Base URL = `Environment.apiUrl`.

## Legend — cache strategy

| Strategy | Meaning |
| --- | --- |
| `cacheFirst` | Return cache instantly, refresh in background (initial screen loads) |
| `networkFirst` | Skip cache, fetch fresh (used on pull-to-refresh / network recovery) |
| `networkOnly` | No cache involvement |
| `none` | Mutation (POST/PUT/PATCH) — not cached |

## Notes & findings

1. **`main_home` is the composite hub** — its dashboard reuses the attendance, announcements, and user-profile cubits, so several endpoints are called both here and from the standalone feature screen.
2. **Two cross-cutting APIs are infrastructure-level:** token refresh (auto via Dio interceptor) and FCM token register/unregister.
3. **File uploads are a two-step flow:** backend presign (`getPresignedUrls`) → direct PUT to AWS S3 (external URL, not in the catalog). Used by example image upload and avatar change.
4. **Dead code:** [`lib/shared/data/repositories/user_profile_repository_impl.dart`](../lib/shared/data/repositories/user_profile_repository_impl.dart) is fully commented out — the live user-profile/avatar logic is in [`lib/features/main_home/data/repositories/user_repository_impl.dart`](../lib/features/main_home/data/repositories/user_repository_impl.dart).
5. **`example/` is a demo/reference feature** (scaffolding template), not part of the production HRM product — `/api/organization/services` and `/api/categories` are placeholder CRUD endpoints.
6. **Logout does not hit the backend** — `signOut` endpoint defined and `AuthApi.logout()` implemented, but the caller is commented out pending backend readiness.
