# Detail Requirement: Sign In

## 1. Use Case Description

As a **platform user with login permission**, I want to **sign in with my email and password** so that **I can access the features authorized for my role**.

**Purpose:** Provide a secure entry point to Exnodes HRM for all authorized users. The sign-in page is the first screen users see and must validate credentials against the system while checking that the user's dynamically assigned role has login permission.

**Target Users:** Any user whose role has been granted login permission. Roles are configurable — they can be created, deleted, and managed through the Role & Permission module (US-004). There are no hardcoded roles.

**Key Functionality:**
- Email and password credential authentication
- "Remember me" for extended session timeout
- Password visibility toggle
- Forgot password link for self-service recovery
- Role-based login permission check
- Redirect to Dashboard on success

---

## 2. User Workflow

**Entry Point:** User opens Exnodes HRM URL in browser. Unauthenticated users always land on the Sign In page.

**Preconditions:**
- User has an active account (created by administrator via permission module)
- User's assigned role has login permission enabled

**Main Flow:**
1. User opens Exnodes HRM URL in browser
2. System displays Sign In page with centered card (dark header with logos/tagline, light form area)
3. User enters email address in email field
4. User enters password in password field
5. User optionally checks "Remember me" checkbox
6. User clicks "Sign In" button (or presses Enter)
7. Button shows loading state; form fields become disabled
8. System validates credentials against stored user records
9. System checks if user's assigned role has login permission
10. System creates user session (extended timeout if "Remember me" checked)
11. System redirects user to Dashboard

**Alternative Flows:**
- **Alt 1 — Forgot Password:** User clicks "Forgot Password" link → navigates to password reset flow (DR-US-001-02)
- **Alt 2 — Show/Hide Password:** User clicks eye icon on password field → password toggles between masked and visible characters
- **Alt 3 — Field Validation on Blur:** User leaves email field → system shows inline error if format invalid; user leaves password field → system shows requirement hint

**Exit Points:**
- **Success:** Redirect to Dashboard
- **Cancel:** N/A — Sign In page is the entry point, no "cancel" destination
- **Error — Invalid credentials:** Toast notification: "Invalid email or password"
- **Error — Account deactivated:** Toast notification: "Your account has been deactivated. Contact your administrator."
- **Error — No login permission:** Toast notification: "You do not have permission to access this system."
- **Error — Account locked:** Toast notification: "Account temporarily locked. Try again in X minutes."
- **Error — Empty fields:** Inline validation below each empty required field

## 6. System Rules

**Business Logic & Backend Behavior:**

- **SR-01:** Credentials validated against stored user records — email lookup followed by password hash comparison
- **SR-02:** After successful credential validation, check if user's assigned role has login permission
- **SR-03:** Failed login attempts tracked per user account (counter increments on each failure)
- **SR-04:** Account locks temporarily after N consecutive failed attempts (N and duration configurable)
- **SR-05:** "Remember me" extends the session timeout duration (session still ends when browser closes, but idle timeout is longer while active)
- **SR-06:** Generic error messages — never reveal whether a specific email exists in the system
- **SR-07:** Multiple concurrent sessions allowed — user can be signed in from different devices/browsers simultaneously
- **SR-08:** Failed attempt counter resets after successful login

**State Transitions:**

```
Unauthenticated → [valid credentials + login permission] → Authenticated (session active)
Unauthenticated → [invalid credentials] → Failed attempt (counter incremented)
Failed attempt (N reached) → [threshold hit] → Account locked (temporary)
Account locked → [lockout period expires] → Unauthenticated (counter reset)
Authenticated → [idle timeout reached] → Unauthenticated (session expired)
Authenticated → [user clicks logout] → Unauthenticated (session cleared)
```

**Dependencies:**
- Authentication service (backend) — credential validation, session management
- Role & Permission module (US-004) — determines which roles have login permission
- Email service — for forgot password flow (DR-US-001-02)

## 7. UX Optimizations

**Usability Considerations:**

- **UX-01:** Auto-focus on email field when page loads — user can start typing immediately
- **UX-02:** Press Enter to submit form from any field (no need to click button)
- **UX-03:** Tab order: Email → Password → Remember me → Sign In → Forgot Password
- **UX-04:** Loading spinner on Sign In button during authentication (prevents double-click)
- **UX-05:** Smooth transition to Dashboard on successful login
- **UX-06:** Toast notifications auto-dismiss after a reasonable duration
- **UX-07:** Email field uses `type="email"` for mobile keyboard optimization
- **UX-08:** Password toggle icon changes between eye-off (masked) and eye (visible) to indicate current state

**Responsive Behavior:**

| Screen Size | Adaptation |
|-------------|------------|
| Desktop (>1024px) | Centered 450px card over full-bleed gradient background (as per Figma) |
| Tablet (768-1024px) | Centered card with slightly reduced padding |
| Mobile (<768px) | Full-width card, background image scales, form fills screen width |

**Accessibility Requirements:**
- [x] Keyboard navigable (Tab, Enter, Space for checkbox)
- [x] Screen reader compatible (labels for email icon, lock icon, eye toggle)
- [x] Sufficient color contrast (WCAG AA compliance)
- [x] Focus indicators visible on all interactive elements
- [x] Toast notifications announced to screen readers (aria-live region)
