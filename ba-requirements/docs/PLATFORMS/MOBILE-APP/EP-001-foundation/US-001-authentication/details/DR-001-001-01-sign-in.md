# Detail Requirement: Sign In

**Detail ID:** DR-001-001-01

**Platform:** MOBILE-APP
**Status:** Draft

## 1. Use Case Description

As an **employee with an active account**, I want to **sign in to the mobile app with my email and password** so that **I can access HRM features on my mobile device**.

**Purpose:** Provide a secure, mobile-optimized entry point to the Exnodes HRM mobile application. The sign-in screen is the first screen users see when launching the app (if not already authenticated) and must validate credentials while providing a smooth mobile experience.

**Target Users:** Any user whose role has been granted login permission. Roles are configurable through the Role & Permission module (WEB-APP US-004). There are no hardcoded roles.

**Key Functionality:**
- Email and password credential authentication
- "Remember me" checkbox for extended session persistence
- Password visibility toggle (eye icon)
- Forgot password link for self-service recovery (OTP-based flow)
- Biometric authentication indicator (future enhancement)
- Secure session token storage on device
- Redirect to Home screen on success

---

## 2. User Workflow

**Entry Point:** User launches the Exnodes HRM mobile app. Unauthenticated users always land on the Sign In screen.

**Preconditions:**
- User has an active account (created by administrator via WEB-APP permission module)
- User's assigned role has login permission enabled
- Mobile device has network connectivity

**Main Flow:**
1. User launches Exnodes HRM mobile app
2. System displays Sign In screen with dark gradient header (logo, "SIGN IN" heading, tagline) and light form card
3. User taps email field (auto-focused on load)
4. System displays email keyboard for email input
5. User enters email address
6. User taps password field
7. User enters password
8. User optionally checks "Remember me" checkbox
9. User taps "Sign In" button
10. Button shows loading state; form fields become disabled
11. System validates credentials against backend authentication service
12. System checks if user's assigned role has login permission
13. System creates user session and stores token in secure device storage
14. System navigates user to Home screen

**Alternative Flows:**
- **Alt 1 — Forgot Password:** User taps "Forgot Password?" link at bottom of screen → navigates to Enter Email screen (DR-001-001-02)
- **Alt 2 — Show/Hide Password:** User taps eye-off icon on password field → password toggles between masked and visible characters; icon changes to eye
- **Alt 3 — Biometric Authentication (Future):** User taps fingerprint icon below Sign In button → initiates biometric authentication flow (future enhancement, not in current scope)

**Exit Points:**
- **Success:** Navigate to Home screen
- **Cancel:** N/A — Sign In screen is the entry point, no "cancel" destination
- **Error — Invalid credentials:** Toast notification: "Invalid email or password"
- **Error — Account deactivated:** Toast notification: "Your account has been deactivated. Contact your administrator."
- **Error — No login permission:** Toast notification: "You do not have permission to access this system."
- **Error — Account locked:** Toast notification: "Account temporarily locked. Try again in X minutes."
- **Error — Network failure:** Toast notification: "Unable to connect. Please check your network connection."

## 5. Acceptance Criteria

**Definition of Done — All criteria must be met:**

**Form & UI:**
- **AC-01:** Sign In screen displays email field (with @ icon), password field (with lock icon and eye toggle), "Remember me" checkbox, "Sign In" button, fingerprint icon, and "Forgot Password?" link
- **AC-02:** Sign In button is disabled (visually distinct) until both email and password fields have input
- **AC-03:** Password field masks characters by default; eye-off icon toggles to eye icon when tapped, revealing password
- **AC-04:** Email field displays email keyboard on focus (with @ and .com keys)
- **AC-05:** Form card has rounded corners (40px radius) and subtle shadow, positioned over dark gradient header

**Validation:**
- **AC-06:** Email field shows inline validation error on blur if format is invalid
- **AC-07:** Sign In button shows activity indicator and form disables while validating

**Authentication:**
- **AC-08:** Valid credentials with login-permitted role navigate user to Home screen
- **AC-09:** Invalid credentials show toast: "Invalid email or password"
- **AC-10:** Deactivated account shows toast: "Your account has been deactivated. Contact your administrator."
- **AC-11:** Role without login permission shows toast: "You do not have permission to access this system."
- **AC-12:** Account locked after N failed attempts shows toast with lockout duration

**Session:**
- **AC-13:** "Remember me" unchecked uses standard session timeout

**Navigation:**
- **AC-16:** "Forgot Password?" link navigates to Enter Email screen for OTP-based password reset
- **AC-17:** Fingerprint icon is visible but non-functional (future enhancement indicator)

**Network & Error Handling:**
- **AC-18:** Network connectivity failure shows toast: "Unable to connect. Please check your network connection."

## 6. System Rules

**Business Logic & Backend Behavior:**

- **SR-01:** Credentials validated against stored user records via shared authentication API (same backend as WEB-APP)
- **SR-02:** After successful credential validation, check if user's assigned role has login permission
- **SR-03:** Failed login attempts tracked per user account (counter increments on each failure)
- **SR-04:** Account locks temporarily after N consecutive failed attempts (N and duration configurable)
- **SR-05:** "Remember me" extends the session token validity duration stored in secure device storage
- **SR-06:** Generic error messages — never reveal whether a specific email exists in the system
- **SR-07:** Multiple concurrent sessions allowed across devices (mobile + web)
- **SR-08:** Failed attempt counter resets after successful login
- **SR-09:** Session token stored using platform-native secure storage (iOS Keychain / Android Keystore)
- **SR-10:** Session persists across app restarts until timeout or explicit logout

**State Transitions:**

```
Unauthenticated → [valid credentials + login permission] → Authenticated (session active)
Unauthenticated → [invalid credentials] → Failed attempt (counter incremented)
Failed attempt (N reached) → [threshold hit] → Account locked (temporary)
Account locked → [lockout period expires] → Unauthenticated (counter reset)
Authenticated → [session timeout reached] → Unauthenticated (token cleared)
Authenticated → [user taps logout] → Unauthenticated (token cleared)
Authenticated → [app backgrounded too long] → Unauthenticated (session expired)
```

**Dependencies:**
- Authentication service (backend) — shared with WEB-APP
- Role & Permission module (WEB-APP US-004) — determines which roles have login permission
- Secure device storage (Keychain/Keystore) — session token persistence

## 7. UX Optimizations

**Usability Considerations:**

- **UX-01:** Auto-focus on email field when screen loads — user can start typing immediately
- **UX-02:** Email keyboard displayed for email field (includes @ and .com keys for faster input)
- **UX-03:** Password autofill support (iOS Keychain / Android Autofill)
- **UX-04:** Activity indicator on Sign In button during authentication (prevents double-tap)
- **UX-05:** Large touch targets for form inputs (minimum 44x44 points)
- **UX-06:** Toast notifications auto-dismiss after 3-4 seconds
- **UX-07:** Haptic feedback on error (optional, device-dependent)
- **UX-08:** Password toggle icon changes between eye-off (masked) and eye (visible) to indicate current state

**Responsive Behavior:**

| Screen Size | Adaptation |
|-------------|------------|
| Standard mobile (375-414px width) | Full-width form card with 30px horizontal padding |
| Small mobile (<375px width) | Reduced padding, scaled typography |
| Large mobile / phablet (>414px width) | Centered form card with max-width, increased padding |

**Accessibility Requirements:**
- [x] Touch targets minimum 44x44 points
- [x] Screen reader compatible (labels for all icons and inputs)
- [x] Sufficient color contrast (WCAG AA compliance)
- [x] Focus indicators visible on all interactive elements
- [x] Toast notifications announced to screen readers (accessibility announcement)
- [x] Keyboard dismissal on tap outside input fields
